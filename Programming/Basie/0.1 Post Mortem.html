Basie 0.1 Post-Mortem Notes.

Positive Stuff.

People.

  • Everyone did a freaking incredible job.

  • Everybody gave more than the expected/requested/paid for hours.

  • The team produced some remarkably clean code in a very short time, given its prior lack of exposure to Django.

  • Distributed collaboration worked much better than expected (much better than many professional/open source teams).

  • It was good that Greg stayed above the programming part, and swooped in when we needed him.

  • Having outstanding expertise on some of the features (i.e. on mail from Blake)

Meetings.

  • Weekly Meetings: I liked the idea of weekly meetings in irc, i think it helped keep things on track and provided an update as to what every one was doing. It also provide a time when you knew every one would be online if you needed information from them.

  • Meetings were run well, with good organization and speed.

  • Meetings were useful. Let's keep them up.

Communication.

  • Mail list: The mail list was a rather effective means of communicating with the rest of the team and seemed to work nicely.

Tools and Supporting Code.

  • Regular, frequent, small commits went well/were a good idea.

  • pyflakes and pep8: Both where a great tool for checking code and style however it would have been nice if we had a script to run both on all files in Basie in one command.

  • Tests: Very useful :)

  • pep8 and pyflakes, mail list and irc

Code Reviews.

  • Code reviews threw the mailing list: Having mailman automatically send an e-mail when there is a commit and reviews replied to it seemed to work very well. It also was a good indication of what was being worked on at the moment and how active things where. The down side was it cluttered up my inbox rather fast.

  • Assigning some one to code reviews: I think code reviews worked well when some one was assigned to just work on doing reviews and leaving the other developers time to code. It lead to better reviews as the person doing the reviews did not worry about having time to finish the code they are working on and could take the time to go threw all the commits.

  • Reviews: Great job with the reviews. It should be maintained next term.

  • Code reviews were very helpful when they explained what was wrong.

  • Blake's code reviews set a great example; it was even greater that so many people followed suit and created a culture of continuous feedback.

  • Code review was really good in theory but the way we were doing it was suck.

Negative

Code Reviews.

  • Code review was really good in theory but the way we were doing it was suck.

  • My biggest disappointment was that the mail application was code reviewed and merged several times into the beta branch. Then, near the end, suddenly it turns out that some of the people weren't satisfied with it. So, I guess, no last minute out-of-the-blue code reviews of the past would help.

  • Code reviews often had "do this 'cause i say so" mentality. We all have different coding styles or even ways at how we approach a problem. Bending to someone's else's wishes without being given a reason does not constitute learning and, that's what army is for.

  • Tempers were strained occasionally during reviews and design discussions (more considerate language should have been used).

People.

  • I wish I wasn't so killed with school/work/etc.

  • We thought that the final meeting would just be a small basie gathering, disappointed that so many outsiders were there.

  • Our distributed team made it hard for the less experienced to ramp up their knowledge; part of this was a hesitance to ask for feedback on the list, but the lack of immediacy hampered them as well. Someone wishes they had more give and take that you get in face to face discussions.

  • Juggling project members. I think that the team that starts working on an application should finish it. It was unfair to take people out cold leaving their work undone and for other people to clean up and finish. So, if this happens in the future, it would be nice to give a little time at least for the person to finish up something that works or is easier to manage.

  • Greg was over-optimistic: set goals that were unachievable given how new most of the dev team was, was too willing to believe that people had finished things and were ready to move on.

  • Greg was too indulgent: should not have given ground on support for multiple version control systems (if those hours had gone into building a wiki, we'd have had one for 0.1).

  • Greg didn't (and doesn't) understand the technology that the project was (and is) using, which meant he could only wave his hands and pontificate when hard decisions needed to be made.

Branches and Merging.

  • We ended up having too many stale branches; we were moving to new features with finishing what we started.

  • svnmerge was a pain

  • svnmerge: The svnmerge.py script was a massive pain to work with and i think it often lead to more work then it saved.

  • Merges: In the last part we had several problems to merge the different branches. I do not know if someone discovered the cause, but perhaps the problem was not to follow the rule: parent --> child branch and child --> parent branch. Next term we probably should merge only trunk --> branch and branch --> trunk.

  • Effort put into svnmerge etc. outweighed value returned.

  • It'll be obvious who this is coming from, but subversion is not the ideal tool for our situation. We're an open source, distributed team using lots of branches and svn does not help us with that workflow. Serious consideration should be put into looking at other systems, how they'd benefit our collaboration, and the drawbacks from devs learning an unfamiliar system and reworking our habits.

Communication.

  • Should have more discussion on list, don't be afraid to ask. Those with experience want to share their knowledge, but it's hard if we don't know the problem. Reviews should not be the time for discovering that you don't understand something.

  • Blog: I found that the blog was not used very effectively and just added a layer of confusion about where to post to.

  • Blog: We could increase the use of the blog. It will be nice if we document the main features of the apps. I had the opportunity to see the development of some apps, but there are others which I don't know almost nothing.

  • A lot of times people seemed to be waffling around without knowing what to do. That could partially be caused by the newness of what we were doing, but I think the newer devs could use more focused direction (akin to class assignments). Having faster iterations on branches could help here as well.

The Release.

  • The release process kind of fell apart, the code freeze never really happened. We should have been more willing to cut code, and refactoring shouldn't be put off.

Tools and Supporting Code.

  • Basie has too much code without tests, and too many bugs got fixed without an accompanying test. Some people complained about spending too much time fixing the tests, but I think that problem stemmed from tests that copied too much instead of making a function, not keeping the tests focused, and also their newness at writing tests.

  • Dependencies: I know the virtualenv works for you but I have to download the dependencies one by one (I can't use buildout). It will be nice if you also document (if they are not in http://basieproject.org/eggs/) where I can find them.

Suggestions

Code Reviews.

  • Reviews are awesome, but we need more reviewers. It may not be easy at first, but gets easier with practice. You learn how to read faster, you get more familiar with the code you're reading.

  • Don't skip the reviews that aren't specifically for you. Reviews are for everyone to learn, I'm wasting my time with involved explanations if only 1 other person is reading it. Reviewers don't want to correct the same problems over and over, it's boring and frustrating.

  • We need a better review process (I despise the email conv on a diff patch)...but this likely won't be fixed until we get a code review tool we can dog feed ourselves.

    • Do you have any ideas on what sorts of things you'ld like from a review tool?

    • I'm impressed with a lot of what Google Code has...but a code review tool is a feature on Greg's radar that I'll likely be involved with, so it's just a waiting game there.

  • More people designated to be review-only. Blake can't do everything ;).

Branches.

  • Have quicker turn around; try to shoot for branches less than 1 week old, max 2 weeks. We'll be better prepared for a release, people will have a better sense of forward progress.

Dogfood!

  • We could increase the use of tickets for the next term.

  • Start using Basie! Self-hosting!
